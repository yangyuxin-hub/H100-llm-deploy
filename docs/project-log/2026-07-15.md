### BFCL adapter 原始响应审计与生成配置纠正

- 原始问题：EvalScope 1.6.1 内置的 `OpenAICompletionsHandler._query_FC` 只把 messages、model、temperature、store 和 tools 传给 OpenAI client，没有转发 EvalScope 的 `max_tokens`、timeout、top-p、seed 和 chat-template 参数。这会让命令行显示的配置与真实 API 请求不一致，也无法仅从 runner 日志区分普通轮次结束、真实漏调用和解析异常。
- 实施修复：新增 `benchmarks/modelcard/bfcl_runtime_patch.py`、`run_evalscope_with_bfcl_patch.py` 和 `summarize_bfcl_audit.py`。补丁仅转发明确配置的请求参数，并逐次写入 `finish_reason`、`content`、`reasoning_content`、`tool_calls`、raw message、usage 和 latency。它不强制 `tool_choice=required`，不把正文后处理为工具调用，不改 BFCL 原生 scorer。
- 首轮审计 smoke `p1-bfcl-memory-audit-smoke-20260715` 曾按临时冻结配置显式传入 `max_tokens=4096`。双模型 `memory_kv` 样例都评为 1/1，但 Qwen 有 22 次 `finish_reason=length`，说明 4K 上限已经影响真实生成；这一轮只能用于诊断，不能作为正式成绩。
