# 2026-07-13 项目日志

## ThinkingCap-Qwen3.6-27B-FP8 官方基线部署

### 目标

在不影响 GPU0,1 上 Qwen3.6-27B-FP8 稳定服务的前提下,停止 GPU2,3 上的 stock Qwen 实验实例,并在同一组实验卡和端口部署 `bottlecapai/ThinkingCap-Qwen3.6-27B-FP8`。

### 部署前检查

远程只读检查确认：

- ThinkingCap 权重位于 `/mnt/nvme0/models/ThinkingCap-Qwen3.6-27B-FP8`,大小约 30G。
- 目录包含 4 个主模型分片、`model-base-aux.safetensors`、`config.json`、`generation_config.json` 和 tokenizer 文件。
- `config.json` 声明 `max_position_embeddings=262144`、`mtp_num_hidden_layers=1`。
- GPU0,1 的稳定容器和 GPU2,3 的 stock Qwen 实验容器均在运行。
- 启动脚本修改前先查看了 stock Qwen 实验容器日志,未发现退出或显存错误。

模型官方页面只要求：

```bash
vllm serve bottlecapai/ThinkingCap-Qwen3.6-27B-FP8
```

因此首次部署采用官方标准解码基线：让 vLLM 自动读取模型配置和采样参数,不额外启用 MTP,不使用 `--language-model-only`,不覆盖 attention backend 或 KV cache dtype。实验环境补充 GPU2,3、TP=2、端口 8001、NUMA 绑定,以及 node1 上已经验证过的 `--max-num-batched-tokens 65536` 安全上限。接入 opencode 后,额外增加 API 兼容所需的 `--enable-auto-tool-choice --tool-call-parser qwen3_coder`。

- `scripts/stop.sh`：新增 `thinkingcap` 停止目标。

结果均为通过。

### 调试记录：本机无法直连 HTTP 端点

执行的命令：

```bash
curl --noproxy '*' http://<node1>:8001/v1/models
```

错误信息：`curl: (7) Failed to connect`。

初步怀疑原因：先发现本机代理变量会把普通 curl 转发到本机代理端口；禁用代理后仍失败,说明当前执行环境只允许 SSH 路径,没有到 node1 服务端口的直接 HTTP 路由。

修复与验证方法：改为通过 SSH 在远程宿主机执行 `curl http://127.0.0.1:8001/...`,并同时使用 `ss -ltnp` 和 Docker 端口映射验证外部监听。

最终结果：远程端点、短推理和 `0.0.0.0:8001` 监听均正常；本机直连失败属于当前执行环境网络限制,不是 vLLM 服务故障。

### opencode 接入与工具调用兼容

本地 `~/.config/opencode/opencode.json` 中原端口 8001 provider 仍指向已经停止的 stock Qwen 实验模型。本次将其替换为：

```text
provider=thinkingcap
model=thinkingcap/thinkingcap-qwen3.6-27b-fp8
context=262144
output=100000
base endpoint=8001/v1
```

稳定 Qwen 继续作为 `model` 和 `small_model`,没有修改默认模型。

首次真实验证命令：

```bash
opencode run --model thinkingcap/thinkingcap-qwen3.6-27b-fp8 \
  "不要解释，只回复：opencode-thinkingcap-ok"
```

错误信息：

```text
"auto" tool choice requires --enable-auto-tool-choice and --tool-call-parser to be set
```

原因：opencode 会随请求提供工具并发送 `tool_choice=auto`,而模型卡的最简 `vllm serve` 命令没有启用服务端 tool parser。普通 curl 对话可以成功,但 opencode 请求会在生成前被 vLLM 拒绝。

修复：ThinkingCap 启动参数只补充以下 Qwen3.6 兼容项,其余模型配置仍由官方权重目录自动读取：

```bash
--enable-auto-tool-choice
--tool-call-parser qwen3_coder
```

按规则执行 `bash scripts/stop.sh thinkingcap` 后重启对应容器,第二次启动 130 秒就绪。再次运行相同 opencode 命令,成功输出 `opencode-thinkingcap-ok`；`opencode models thinkingcap` 也能列出正确模型 ID。

### 学习记录

- “ThinkingCap 减少任务所需 thinking tokens”和“MTP 提高每秒输出 tokens”是两种独立收益。首次部署关闭 MTP,才能先测清微调本身的 token 效率。
- 不显式传入 sampling、attention 和 KV 参数时,vLLM 会分别从模型 `generation_config.json` 和运行时硬件能力选择默认值；启动日志是确认实际生效配置的重要证据。
- 完整多模态加载下 KV cache 容量低于此前 text-only + FP8 KV 的 stock Qwen 实验实例,后续 benchmark 必须记录这一配置差异。
- OpenAI-compatible 端点能完成普通聊天,不代表 agent 客户端一定兼容；必须用真实客户端验证 `tools` 和 `tool_choice` 请求路径。

## ThinkingCap 启用 MTP3

### 配置

模型 `config.json` 声明：

```text
mtp_num_hidden_layers=1
mtp_use_dedicated_embeddings=false
```

ThinkingCap FP8 模型卡说明 MTP head 保持 bf16,并报告约 3.35 的平均接受长度。为对齐该测试深度,配置 3 个 speculative tokens：

```bash
--speculative-config '{"method":"mtp","num_speculative_tokens":3}'
```

最初使用兼容别名 `qwen3_next_mtp` 启动时,vLLM 0.24.0 提示该名称已弃用并自动归一化为 `method='mtp'`。当前持久配置已改用新的 `mtp` 方法名；正在运行的引擎也已确认其内部配置为 `SpeculativeConfig(method='mtp', num_spec_tokens=3)`。

### 启动结果

按规则先执行：

```bash
bash scripts/stop.sh thinkingcap
bash scripts/start_thinkingcap_docker.sh
```

停止后 GPU2,3 显存降至 1 MiB,GPU0,1 稳定端点保持健康。ThinkingCap MTP 首次启动耗时 250 秒,新增长耗时来自 MTP/eagle head 的首次 graph 编译。

启动日志确认：

- 检测到 MTP 模型并与 target model 共享 embedding 和 `lm_head`。
- 加载 MTP 权重后创建独立 `eagle_head` 编译缓存。
- Docker 实际参数包含 `num_speculative_tokens=3`。

### 真实生成验证

使用 27 prompt tokens、最多 512 output tokens 的中文列表请求,实际生成 496 tokens。vLLM 指标：

```text
Mean acceptance length: 2.40
Accepted throughput: 6.06 tokens/s
Drafted throughput: 12.97 tokens/s
Accepted: 290 tokens
Drafted: 621 tokens
Per-position acceptance rate: 0.676, 0.440, 0.285
Avg Draft acceptance rate: 46.7%
```

该结果证明 speculative decoding 确实参与生成,而不只是启动参数存在。单次中文列表请求的 2.40 接受长度低于模型卡约 3.35,不能直接判断模型异常；接受率会随 workload、采样和输出分布变化,需要在正式 benchmark 中使用相同数据集比较。

启用后 GPU2,3 显存各约 76749 MiB,比标准解码约 73197 MiB 增加约 3.55 GiB,主要来自 MTP head、额外编译图和运行时工作区。

### opencode 复测

MTP 启用后运行：

```bash
opencode run --model thinkingcap/thinkingcap-qwen3.6-27b-fp8 \
  "不要解释，只回复：mtp-opencode-ok"
```

最终成功返回 `mtp-opencode-ok`,说明 MTP 与 `qwen3_coder` tool parser、opencode 请求链兼容。

复测期间曾出现 `opencode: command not found`。检查 npm 日志后确认,另一个并行全局更新正在将 opencode 1.17.16 升级到 1.17.18；当时正处于旧入口已移动、Linux 平台包仍在下载的原子替换窗口。安装进程随后自行成功完成,没有重复安装；`opencode --version` 和模型列表恢复正常。

## ThinkingCap MTP3 在线吞吐测试

### 测试口径

在不重启或修改容器的前提下,对 GPU2,3 上当前运行的 ThinkingCap 实验服务执行在线 benchmark。测试前通过 Docker inspect 确认实际配置为 TP=2、`max-num-batched-tokens=65536`、MTP 3 draft tokens；Docker CLI 仍显示兼容别名 `qwen3_next_mtp`,vLLM 内部会归一化为 `mtp`。

使用 `vllm bench serve` 经 `/v1/completions` 测试 random dataset：固定输入 500 tokens、强制输出 512 tokens、`ignore-eos`、`request-rate=inf`,依次扫描并发 1/4/8/16/32。每档正式请求数为并发数的 16 倍,另加 2 个 warmup 请求；seed 固定为 42,未覆盖服务端采样参数。

### 结果

| 并发 | 请求数 | output tok/s | total tok/s | TTFT 中位/P99 (ms) | TPOT 中位/P99 (ms) | MTP 接受率 | 接受长度 |
|---:|---:|---:|---:|---:|---:|---:|---:|
| 1 | 16 | 176.9 | 349.7 | 51.4 / 53.7 | 5.08 / 11.09 | 49.9% | 2.50 |
| 4 | 64 | 629.6 | 1244.5 | 68.0 / 145.0 | 5.39 / 15.30 | 56.0% | 2.68 |
| 8 | 128 | 1164.5 | 2301.8 | 84.8 / 437.9 | 6.21 / 12.64 | 58.9% | 2.77 |
| 16 | 256 | 1931.8 | 3818.2 | 89.0 / 482.7 | 7.33 / 19.35 | 56.1% | 2.68 |
| 32 | 512 | 2807.4 | 5549.1 | 104.1 / 779.4 | 10.08 / 26.11 | 58.2% | 2.74 |

976 个正式请求全部成功。并发 32 的输出吞吐相对并发 16 增长 45.3%,说明尚未形成硬平台,但 TPOT 中位增加 37.6%、TTFT P99 增加 61.5%,边际收益和长尾延迟代价已经明显。交互延迟优先时并发 16 更均衡,批处理吞吐优先时当前实测最高点是并发 32。

测试后确认 8000/8001 两个端点和两个容器均健康,GPU2,3 回到空闲,ThinkingCap 最近 10 分钟日志未发现 OOM、error、exception 或 traceback。原始 JSON 和完整说明保存在 `logs/bench/thinkingcap-20260713-mtp3/`。

### 学习记录

- MTP 接受率不是模型的固定属性。本轮 random token workload 的接受率约 50%–59%、平均接受长度 2.50–2.77,不能直接外推到真实 coding/agent 请求。
- output tok/s 随并发提高不代表延迟也更好；容量规划必须同时看 aggregate throughput、TTFT P99 和 TPOT P99。
- 历史 stock Qwen benchmark 使用 TP4+MTP2,本轮 ThinkingCap 使用 TP2+MTP3；配置不同,绝对吞吐不能直接用于判断 ThinkingCap 相对 stock Qwen 的模型收益。

## 稳定 Qwen3.6-27B-FP8 同口径吞吐对比

### 测试口径

用户所称的“Qwen32B 稳定服务”经远程 `/v1/models` 确认为 `Qwen3.6-27B-FP8`,运行在 GPU0,1、TP=2、端口 8000。Docker 实际参数为 `FLASH_ATTN`、FP8 KV、MTP=3、`max-num-seqs=16`、`max-num-batched-tokens=65536`；没有重启或修改稳定容器。

复用 ThinkingCap MTP3 的 random workload：输入 500 tokens、输出 512 tokens、`ignore-eos`、`request-rate=inf`、seed=42、并发 1/4/8/16/32，每档正式请求数为并发×16，另加 2 个 warmup 请求。976 个正式请求全部成功。

### 结果

| 并发 | 请求数 | output tok/s | total tok/s | TTFT 中位/P99 (ms) | TPOT 中位/P99 (ms) | MTP 接受率 | 接受长度 |
|---:|---:|---:|---:|---:|---:|---:|---:|
| 1 | 16 | 244.6 | 483.5 | 68.1 / 78.2 | 3.89 / 4.91 | 75.6% | 3.27 |
| 4 | 64 | 741.6 | 1465.8 | 96.4 / 165.1 | 4.68 / 14.14 | 73.1% | 3.19 |
| 8 | 128 | 1399.8 | 2766.9 | 98.9 / 251.0 | 5.36 / 7.19 | 72.9% | 3.19 |
| 16 | 256 | 2181.9 | 4312.7 | 107.1 / 1422.4 | 6.57 / 9.92 | 73.0% | 3.19 |
| 32 | 512 | 1868.7 | 3693.5 | 5263.4 / 6279.2 | 6.74 / 14.41 | 71.1% | 3.13 |

### 与 ThinkingCap MTP3 的观察

- 并发 1/4/8/16，稳定 Qwen 的 output tok/s 分别是 ThinkingCap 的约 1.38x、1.18x、1.20x、1.13x；当前 workload 下 stock Qwen 的 decode 吞吐更高。
- 并发 32 时 Qwen output tok/s 降至 1868.7，而 ThinkingCap 为 2807.4；Qwen 的 TTFT 中位升到 5.26s。主要原因是稳定服务明确设置了 `max-num-seqs=16`，并发 32 的请求发生排队，不能把这一档直接解释为模型能力差异。
- Qwen MTP 接受长度为 3.13–3.27，明显高于本轮 ThinkingCap 的 2.50–2.77；这解释了 Qwen 在低至中并发下较高的 decode 吞吐，但接受率仍会随 workload 变化。
- 两个服务都是 TP=2、MTP=3，但显存布局、服务启动默认项和 `max-num-seqs` 不完全相同；本结果可作为当前生产配置对比，不是严格的模型 A/B 结论。8000/8001 复核均健康，Qwen 容器日志无 OOM、error、exception 或 traceback。

学习价值：`max-num-seqs` 不只是容量配置，也会直接改变高并发 benchmark 的排队延迟和 aggregate throughput。比较模型前必须先对齐调度上限，否则并发 32 的结果会把调度瓶颈误判为模型速度。

## 稳定 Qwen 长上下文扫描（独立 seed 清洁版）

### 测试说明

为对比截图中的 ThinkingCap 长上下文结果，稳定 Qwen 使用相同输入长度 `8K/32K/64K/120K/140K`、统一输出 512 tokens、并发 1/4。每档分别使用 8/16 个正式请求，另加 1 个 warmup，`request-rate=inf`、`ignore-eos`。

稳定 Qwen 开启了 prefix caching，因此不能用固定 seed 按长度递增扫描；否则后续长度会复用前面长度的随机前缀。清洁版为每个输入长度、每个并发组使用独立 seed，并严格串行执行，避免跨档缓存污染。10 档共 240 个正式请求全部成功。

| 输入 | TTFT 中位/P99 | TPOT 中位/P99 | output tok/s |
|---:|---:|---:|---:|
| 8K | 0.42s / 0.43s | 3.61 / 11.15ms | 179.6 |
| 32K | 1.73s / 1.75s | 4.67 / 5.06ms | 133.5 |
| 64K | 3.96s / 3.99s | 3.83 / 5.97ms | 89.8 |
| 120K | 9.07s / 9.08s | 4.11 / 4.93ms | 50.3 |
| 140K | 11.29s / 11.32s | 4.41 / 5.23ms | 41.5 |

### 并发 4 结果

| 输入 | TTFT 中位/P99 | TPOT 中位/P99 | total tok/s |
|---:|---:|---:|---:|
| 8K | 0.48s / 1.22s | 5.90 / 8.32ms | 9.30K |
| 32K | 1.80s / 5.14s | 13.26 / 18.14ms | 15.28K |
| 64K | 5.95s / 11.77s | 20.07 / 40.66ms | 15.41K |

| 140K | 19.16s / 32.65s | 47.19 / 170.86ms | 12.81K |

### 与截图中 ThinkingCap 的对比

- 单并发 TTFT：Qwen 与 ThinkingCap 在 8K/32K 接近；64K/120K/140K，Qwen 分别约慢 0.30s、1.14s、1.50s。
- 并发 4 的 total tok/s：Qwen 在 8K/32K/64K 分别为 9.30K/15.28K/15.41K，高于 ThinkingCap 的 6.84K/14.03K/15.51K；120K/140K 则为 13.67K/12.81K，低于 ThinkingCap 的 13.97K/13.94K。
- Qwen 在长输入并发 4 时 TPOT 和 TTFT 长尾增长更明显，140K 的 TPOT P99 达 170.86ms；超长请求应与普通交互请求隔离队列。
- 这是当前两个服务配置的实测对比，不是纯模型 A/B：Qwen 明确启用了 prefix caching、`max-num-seqs=16` 和 MTP3，ThinkingCap 的启动默认项不同。清洁版仅消除了本次 benchmark 的跨档 prefix cache 污染。

原始结果在测试期间保存在 node1 容器 `/tmp/qwen-long-clean/`；本地批量 `scp` 归档曾遇到 SSH connection closed，汇总数据已记录在本节。

## ThinkingCap 基础能力 benchmark（MMLU-Pro pilot）

### 执行与结果

### 学习记录

- 能力回归测试必须保存每题原始输出、抽取答案和题目 ID；接口返回为空、思维预算耗尽、解析提前停止都要作为独立的有效性指标记录。

### 数据与 harness 边界

已核对公开数据源：GPQA 是 gated dataset，需要用户先在 Hugging Face 接受条款并登录；MMLU-Redux 和 C-Eval 包含大量 config；LiveCodeBench、RealWorldQA、AA-LCR、System-prompt 和 Claw-Eval 需要专用执行或裁判 harness，不能只下载 parquet 就宣称评测准备完成。

检查当前 `lm-eval 0.4.12` 后发现，C-Eval、LongBench v2、ARC-Easy、CommonsenseQA、OpenBookQA 和 SciQ 等内置任务使用 `multiple_choice` loglikelihood。这与 ThinkingCap 模型卡的“模型生成答案后匹配最后一个选项字母”不是同一口径，也未必适用于当前 OpenAI Chat Completions 服务。因此 manifest 把这些任务标为“生成式适配器”，后续必须先完成 adapter，再做正式 A/B。

### 学习价值

- “数据集名称相同”不代表评测口径相同；prompt、输出方式、答案抽取和评分器任何一项不同都会改变分数。
- ThinkingCap 的核心 KPI 是质量保持、逐题 thinking-token reduction 和 time-to-correct，三者必须成组报告。
- 官方 Reduction 不是用表中两个总体均值直接相除，而是先在同一道题内跨 seed 求均值，再计算逐题 `(Base-Cap)/Base`，最后对共享题平均。
- 域外、域内和安全套件的分数含义不同，应分别计算宏平均，不能混成一个项目总分。
