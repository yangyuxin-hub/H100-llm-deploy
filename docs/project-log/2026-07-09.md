## 2026-07-09 Qwen GPU2,3 实验实例 120K 长上下文 prefill 延迟测试

### 目标

测试 `qwen3.6-27b-fp8-gpu23` 在 120K prompt tokens 下的单请求 prefill 延迟,用于理解长上下文请求的 TTFT 与 prompt 处理吞吐。

### 测试前状态

远程 node1 上执行:

```bash
cd /root/llm-deploy && bash scripts/status.sh
```

结果:

- `qwen3.6-27b-fp8-gpu23` 就绪,端口 8001,GPU2,3。
- `qwen3.6-27b-fp8` 就绪,端口 8000,GPU0,1。
- GPU2,3 测试前显存约 `72041 MiB / 81559 MiB`。
- `/v1/models` 返回 `max_model_len=262144`,满足 120K prompt 测试。
- vLLM 日志显示实验实例 KV cache 约 `2,816,558 tokens`,262144 长度请求最大并发约 `10.74x`。

### 测试方法

先用服务自带 tokenizer 校准 prompt 构造:

```bash
POST http://127.0.0.1:8001/tokenize
{"model":"qwen3.6-27b-fp8-gpu23","prompt":" x" * 120000}
```

返回 `count=120000`。`" x"`, `" y"`, `" z"` 均为单 token,因此使用三组不同重复 token,避免重复同一 prompt 被 prefix cache 影响。

正式请求使用 OpenAI-compatible completions 流式接口:

```json
{
  "model": "qwen3.6-27b-fp8-gpu23",
  "prompt": "<120000 prompt tokens>",
  "max_tokens": 1,
  "temperature": 0,
  "stream": true
}
```

客户端记录从发起 HTTP POST 到第一个非空流式 text token 的时间。因为 `max_tokens=1`,该 TTFT 基本可视为本次长 prompt 的 prefill 延迟 + API/调度开销 + 第一个 decode step。

### 测试结果

| run | prompt | prompt tokens | TTFT / prefill 近似 | total latency | 输出首 token |
|---|---:|---:|---:|---:|---|
| 1 | `" x" * 120000` | 120000 | 26.299 s | 26.299 s | `" x"` |
| 2 | `" y" * 120000` | 120000 | 25.879 s | 25.879 s | `" y"` |
| 3 | `" z" * 120000` | 120000 | 25.855 s | 25.855 s | `" z"` |

统计:

- 平均 TTFT:约 `26.011 s`
- 最快:约 `25.855 s`
- 最慢:约 `26.299 s`
- 按 120000 prompt tokens / TTFT 粗算,端到端 prompt 处理吞吐约 `4.56K-4.64K tokens/s`

vLLM 10 秒窗口日志显示:

```text
Avg prompt throughput: 12000.4 tokens/s
Avg prompt throughput: 11999.1 tokens/s
Avg prompt throughput: 11999.4 tokens/s
Prefix cache hit rate: 0.0%
```

Prometheus `/metrics` 验证:

- `vllm:request_prompt_tokens_count` 从历史累计看共有 5 个请求,其中 3 个落在 `100000-200000` prompt token bucket。
- `vllm:request_prompt_tokens_sum=360034`,与三次 120K 测试加少量短请求一致。
- `vllm:time_to_first_token_seconds` 中三次 120K 请求落在 `20-40s` bucket。

### 测试后状态

测试后再次执行:

```bash
cd /root/llm-deploy && bash scripts/status.sh
```

结果:

- `qwen3.6-27b-fp8-gpu23` 仍就绪。
- `qwen3.6-27b-fp8` 仍就绪。
- GPU2,3 显存变为约 `80987 MiB / 81559 MiB`,高于测试前的约 `72041 MiB`。这说明 120K 长上下文请求后,vLLM/CUDA 相关内存池或 KV/cache 分配有驻留增长,不代表当前仍有运行中请求。

### 学习记录

- 长上下文 prefill 的用户可感知延迟主要体现在 TTFT,用流式请求和 `max_tokens=1` 可以较直接地测出。
- vLLM 日志中的 `Avg prompt throughput` 是滑动窗口吞吐,不等同于单请求端到端吞吐；客户端 TTFT 包含调度、HTTP、首个 decode step 等开销。
- prefix cache 会显著影响重复 prompt 的测量结果,因此多轮长上下文 benchmark 应避免使用相同长前缀。

### 2026-07-09 补充:瓶颈初步判断

再次使用 `" k" * 120000` 发起 120K prompt tokens / `max_tokens=1` 流式请求,TTFT 为 `25.846 s`。同时使用 `nvidia-smi dmon -i 2,3 -s pucm -d 1 -c 40` 采样 GPU2,3:

- 请求活跃阶段 GPU2,3 的 SM 利用率长期为 `99-100%`。
- 功耗活跃阶段约 `350-430 W`。
- `dmon` 的显存带宽利用率多数在个位数到二十几,少数采样点到约 `40%`。
- 显存驻留约 `80987 MiB / 81559 MiB`。

初步判断:

- 120K prefill 阶段主要瓶颈在 GPU 计算/模型 kernel 路径,不是 HTTP、CPU 调度、PCIe/网络或纯 HBM 带宽。
- `qwen3_5_text` 配置为 64 层,其中 16 层 `full_attention`,48 层 `linear_attention`。120K 下 full attention 的 `O(n^2)` 计算和 linear attention/Mamba 相关自定义 kernel 都会贡献明显开销。
- 当前启动参数 `--max-num-batched-tokens=65536` + `--enable-chunked-prefill` 会把 120K prompt 至少切成两段 prefill,但从 SM 长时间满载看,它不是主要空转瓶颈。

#### 重要发现:thinking mode 会影响用户可见内容

Qwen3.6 默认 thinking mode。普通 OpenAI Chat 请求如果只给 `max_tokens=512`,可能 512 tokens 都消耗在 reasoning 中,最终 `content=null` 或流式 `content` 不可见。补测:

```json
{
  "chat_template_kwargs": {"enable_thinking": false}
}
```

后,短问答能直接返回:

```text
1+1等于2。
```

因此面向普通用户体验的测试应区分:

- thinking mode:适合需要展示/使用推理过程的任务,需要更大 `max_tokens`
- non-thinking mode:适合日常问答、文档总结、工具型响应,首响和最终答案体验更好

#### thinking mode 初测

测试场景:

| case | 输入 | 输出上限 |
|---|---:|---:|
| chat_short | 约 33 prompt tokens | 512 |
| doc_32k_qa | 约 32K prompt tokens | 512 |
| doc_120k_qa | 约 120K prompt tokens | 512 |
| long_output | 约 43 prompt tokens | 2048 |

flashinfer 结果:

| case | 结果 |
|---|---|
| chat_short | e2e `6.783s`,但流式可见内容为 0,512 tokens 主要在 reasoning |
| doc_32k_qa | e2e `6.634s`,但流式可见内容为 0,512 tokens 主要在 reasoning |
| doc_120k_qa | 客户端约 `10.631s` 后结束,未拿到完整 usage/metrics |
| long_output | HTTP 500,EngineCore 退出 |

flashinfer 崩溃日志关键错误:


## 2026-07-09 GPU2,3 实验实例默认切换到 FLASH_ATTN

### 背景

面向用户体验的测试显示,在 GPU2,3 实验实例 `qwen3.6-27b-fp8-gpu23` 上:

- 120K 冷 prefill: `FLASH_ATTN` 约 `9.1s`,原 `flashinfer` 约 `26s`。
- non-thinking 32K 文档问答: `FLASH_ATTN` TTFT `1.970s`,总耗时 `2.557s`; `flashinfer` TTFT `3.318s`,总耗时 `4.208s`。

- `QWEN_FP8_EXP_ATTENTION_BACKEND`:实验实例 `qwen3.6-27b-fp8-gpu23`,GPU2,3,端口 8001,改为 `FLASH_ATTN`。

## 2026-07-09 清理 opencode 中的 Agents-A1 残留

### 背景

用户发现 `opencode models` 里仍显示 `agents/agents-a1-fp8`。远程 8001 当前实际运行的是 `qwen3.6-27b-fp8-gpu23`,不是 Agents-A1,因此这是本地 opencode 配置残留。

### 根因

本地 `~/.config/opencode/opencode.json` 中:

- `small_model` 仍指向 `agents/agents-a1-fp8`
- `provider.agents` 仍注册为 `http://10.16.11.24:8001/v1`

这会让 opencode 把 8001 显示成 Agents-A1,与当前远程服务不一致。

### 处理

保留 Qwen provider,删除 `provider.agents`,并把:

```json
"small_model": "qwen/qwen3.6-27b-fp8"
```

### 验证

`opencode models` 不再显示 `agents/agents-a1-fp8`,只保留本地 H100 的:

