# ThinkingCap MTP 接受率改动前后分析

## 结论

ThinkingCap 配置对齐后观察到的 MTP 接受率跃升，主要原因不是 NUMA、custom all-reduce 或容器参数，而是默认采样温度由模型自带的 `temperature=1.0` 改成了与稳定 Qwen 一致的 `temperature=0.6`。

同温度 `1.0` 的前后对照显示，新运行配置让输出吞吐提升约 4%，但 MTP 接受率没有提升。保持新运行配置不变，只把温度从 `1.0` 降到 `0.6`，接受率从 55.44% 上升到 72.43%，平均接受长度从 2.66 上升到 3.17。

因此应把本次收益拆成两部分：

- 采样参数变化主要提高 MTP 接受率，并进一步降低 TPOT、提高输出吞吐。
- NUMA、custom all-reduce、显式 GPU 分配、FlashInfer sampler 等运行配置主要降低系统开销，改善吞吐和延迟稳定性。

## 对照方法

三次测试都使用 ThinkingCap、MTP3、GPU2,3、TP=2，以及相同的 random workload：输入 500 tokens、固定生成 512 tokens、`ignore-eos`、并发 16、无限到达率、258 个正式请求、seed 42。

| 组别 | 运行配置 | temperature | 用途 |
|---|---|---:|---|
| A：改动前 | 2026-07-13 parity 配置 | 1.0（模型默认） | 历史基线 |
| B：改动后 | 与稳定 Qwen 对齐后的配置 | 1.0（请求显式指定） | 隔离运行配置影响 |
| C：改动后 | 与稳定 Qwen 对齐后的配置 | 0.6（请求显式指定） | 隔离采样温度影响 |

A 与 B 的温度和 workload 相同，差异主要是运行环境；B 与 C 的运行环境完全相同，只改变温度。这比直接比较改动前与改动后的服务默认值更能判断因果。

## 核心结果

| 指标 | A：改动前 T=1.0 | B：改动后 T=1.0 | C：改动后 T=0.6 |
|---|---:|---:|---:|
| 成功请求 | 258/258 | 258/258 | 258/258 |
| MTP 接受率 | 58.54% | 55.44% | **72.43%** |
| MTP 平均接受长度 | 2.76 | 2.66 | **3.17** |
| 第 1/2/3 个 draft 位置接受率 | 72.60% / 57.25% / 45.78% | 69.43% / 53.92% / 42.98% | **84.76% / 71.96% / 60.55%** |
| 输出吞吐 | 1872.73 tok/s | 1947.91 tok/s | **2264.57 tok/s** |
| 总吞吐 | 3701.58 tok/s | 3850.16 tok/s | **4476.07 tok/s** |
| TTFT 中位 | 98.96 ms | **93.48 ms** | 94.36 ms |
| TTFT P99 | 627.59 ms | 603.02 ms | **457.11 ms** |
| TPOT 中位 | 7.58 ms | 7.32 ms | **6.54 ms** |
| TPOT P99 | 19.94 ms | 18.92 ms | **13.54 ms** |
| 测试耗时 | 70.54 s | 67.81 s | **58.33 s** |

### 运行配置本身的影响：A → B

- 输出和总吞吐提升约 4.0%。
- TTFT 中位降低约 5.5%，TPOT 中位降低约 3.5%。
- 接受率反而降低 3.10 个百分点，平均接受长度降低约 0.10。

这说明运行配置对齐确实有性能收益，但不能解释接受率上涨。A 与 B 不是重复多次取均值，因此接受率的 -3.10pp 不宜解读成配置必然导致退化；它足以说明没有出现十几个百分点的正向提升。

### 采样温度的影响：B → C

- MTP 接受率增加 16.99 个百分点，相对提升约 30.6%。
- 平均接受长度从 2.66 增至 3.17，提升约 19.1%。
- 输出吞吐增加约 16.3%，TPOT 中位降低约 10.6%。
- 三个 draft 位置的接受率全部显著提高，而不是单个位置偶然波动。

这组只改变温度的测试，直接证明 `temperature=0.6` 是接受率立即上升的主因。

## 为什么温度会影响 MTP 接受率

MTP head 会先提出后续 token，主模型 verifier 再根据目标采样分布决定是否接受。温度越低，目标分布通常越尖锐，高概率 token 更集中；MTP head 提出的高概率 token 因而更容易与 verifier 的选择一致。

`temperature=1.0` 的随机性更高，即使 draft token 本身合理，也更容易因为 verifier 从较分散的分布中选到其他 token 而被拒绝。这里三个 draft 位置都随温度降低而提高，符合这一机制。

平均接受长度可按 `1 + accepted_tokens / num_drafts` 理解。以每次生成 512 tokens 粗略估算：

- A 的平均长度 2.76，需要约 `512 / 2.76 = 186` 轮 verifier 推进。
- C 的平均长度 3.17，需要约 `512 / 3.17 = 161` 轮。

即 verifier 轮数约减少 13%，这与 TPOT 和吞吐改善方向一致。实际吞吐还会同时受到 batch 调度、请求完成节奏和运行环境的影响，不能只由该估算精确推出。

## 改动项的作用分类

| 改动 | 对 MTP 接受率的预期影响 | 本次证据 |
|---|---|---|
| `temperature: 1.0 → 0.6` | **直接且显著** | 同配置消融：55.44% → 72.43% |
| custom all-reduce 开启 | 主要影响 TP 通信开销 | 同温度前后只有约 4% 吞吐收益，接受率未升 |
| vLLM 原生 NUMA node 1 绑定 | 主要影响 CPU/GPU 数据路径和抖动 | 更可能改善延迟，不改变模型 logits |
| 显式 GPU DeviceIDs 2,3 | 提高设备隔离和可复现性 | 不应直接改变 draft/verifier 一致性 |
| FlashInfer sampler | 主要影响采样 kernel 开销 | 可能贡献吞吐，不足以解释接受率跃升 |
| `SYS_NICE`、16 GiB shm | 运行稳定性和共享内存条件 | 不直接改变接受判定分布 |
| reasoning parser / thinking 输出分离 | 只影响 API 输出字段组织 | `/v1/completions` random benchmark 不受影响 |
| MTP3、TP2、FP8 KV、FLASH_ATTN、batch 上限 | 前后保持一致 | 不是本次立即上涨的变量 |

## 如何看线上日志

vLLM 日志中的 MTP 指标通常是短时间窗口值。窗口内请求很少时，接受率会在 50%～90% 间明显波动，不应拿单个 10 秒窗口判断长期变化。

本次配置重启后的累计指标约为：接受率 74.76%、平均接受长度 3.24；三个位置约为 89.59%、74.31%、60.37%。它与独立 `temperature=0` 的 64 请求测试（74.69%、3.24）接近，但不能与 258 请求的 `temperature=0.6` 测试混成同一口径。

## 使用建议

- 当前生产默认 `temperature=0.6, top_p=0.95, top_k=20` 可以保留，它兼顾了模型推荐采样和较高 MTP 接受率。
- 不建议只为追求接受率把所有请求固定到 `temperature=0`；它会改变输出多样性和任务质量，需要按真实 coding/agent 任务评估。
- 后续比较配置时，应在请求中显式传入 temperature，并至少重复 3 个 seed，报告均值与波动范围。
- 观察服务时优先使用累计 `/metrics`，短窗口日志只用于发现趋势或异常。
