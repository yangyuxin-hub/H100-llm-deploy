# 附录 B:微调数据测评流程

本附录是"微调后模型如何回归评测"的流程骨架。当前仓库只做现成模型权重评测,本页为后续自己做微调数据测评预留,具体命令待训练框架和微调数据确定后补充。

## 目标

验证微调后模型:

1. **能力保持**:通用能力没有显著回退(用 P0 七项回归)。
2. **微调增益**:微调目标能力有可度量提升(用专属 benchmark)。
3. **可比性**:与 stock 基线在同题、同配置、同协议下对比。

## 适用范围

- 被测模型:用户基于 `Qwen3.6-27B-FP8` 微调( SFT / LoRA / 全参)得到的权重。
- 基线模型:`Qwen3.6-27B-FP8`(stock,部署在 8000 端点)。
- 微调模型部署在 8001 端点(复用现有 `start_*_docker.sh`,替换权重路径)。

## 流程骨架

```text
1. 微调数据准备
   ├── 数据集版本号、checksum 冻结
   ├── 记录数据来源、license、难度分布、覆盖度
   └── 去重、毒性、质量筛检记录

2. 训练执行
   ├── 训练框架、超参、GPU 配置记录
   ├── checkpoint 路径与版本
   └── 训练日志归档

3. 权重合并与转换
   ├── LoRA 合并到基线权重(如适用)
   ├── 转换为 vLLM 可加载格式(FP8/safetensors)
   └── 校验权重 SHA256

4. 部署验证
   ├── 复用 start_*_docker.sh 部署到 8001
   ├── smoke:确认端点可服务、模型名正确
   └── 验证 chat template 一致性(若改了模板,先跑 TAU2 smoke)

5. 能力保持评测(回归)
   ├── 复用 P0 七项,同 seed=42、同协议
   ├── 与 stock 基线 A/B
   └── 重点看是否回退超过判收阈值

6. 微调增益评测
   ├── 针对微调目标的专属 benchmark
   ├── 与 stock 基线对比提升幅度
   └── 报告增益是否显著(考虑多 seed 波动)

7. 回归对比与判收
   ├── 配对 bootstrap 报告差值置信区间
   ├── 写卡到 benchmark-comparison.md(新增"微调版"列)
   └── 记录微调数据版本、训练配置、checkpoint 路径
```

## 待决策问题

开始微调数据测评前,需要先确定:

1. **测什么**:只测"微调后模型能力保持",还是同时测"训练数据本身的质量"(去重率/毒性/难度分布/覆盖度)?
2. **判收标准**:能力下降多少算通过?建议按维度分别设阈值(如指令遵循不允许下降,长上下文允许 2%),还是统一阈值?
3. **回归范围**:全量重跑 P0 七项(贵),还是抽子集(如 MMLU-Pro 280 题 pilot)?
4. **多 seed**:是否报告多 seed 均值与方差?当前现成模型评测是单 seed=42。
5. **专属 benchmark**:微调目标对应哪个 benchmark?(如代码微调用 LiveCodeBench,Agent 微调用 TAU2)
